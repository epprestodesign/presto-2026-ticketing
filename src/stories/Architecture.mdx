# Architecture & Conventions

The machine-oriented map of this library. If you're an **AI agent** extending the
system or **building the localized prototype**, read this first — it tells you
**where things live, what to reuse, the conventions to match, and how the booking
modes map across components** so new work is indistinguishable from existing work.

For the stage-by-stage journey requirements (all flows, all edge cases), read
**Getting Started → User Journey & Build Spec**.

---

## Reading the system efficiently

1. **Find the surface** in the sidebar — primitives (`Components → …`,
   `Foundations`) or experience flows (`Landing Page … Manage Booking`). See
   **Introduction** for the category map.
2. **Locate the source** — every story maps ~1:1 to a component (tables below).
3. **Reuse before rebuilding** — check *Shared building blocks*; most new screens
   compose existing pieces.
4. **Match the conventions** — tokens, naming, story shape, and the **mode
   cheat-sheet** below.

## Booking modes — the naming cheat-sheet (READ THIS)

There are three conceptual modes (**reserve · hold · reservations**), but the
prop **name and accepted values differ per component**. Passing the wrong string
silently renders the wrong flow. Canonical mapping:

| Component | Prop | Single stay | Group/team block | Multiple reservations |
| --- | --- | --- | --- | --- |
| `LandingPage`, `BookingWidget` | `mode` | `reservations`¹ | `group` | `reservations`¹ |
| `CheckoutPage` | `mode` | `reservation` | `group` | `reservations` |
| `CartReview`, `ConfirmationPage` | `mode` | `reserve` | `hold` | `reservations` |
| `HotelDetailPage` | `roomsFlow` | `reserve` | `group` | `reserve` |
| `RoomsCarousel` | `flow` | `reserve` | `group` | `reserve` |
| `HotelCardReserve` | `availability` | `available` / `unmatched` / `unavailable` | — | — |
| `HotelCardGroup` | `flow` + `availability` | `flow="reserve"` | `flow="group"`, `matches`/`partial`/`unavailable` | — |
| `StepReviewOrder` | `mode` | `reserve` | `hold` | `reserve` |
| `StepReviewReservation` | `flow` | `reserve` | `group` | `reserve` |
| `StepContactInfo` | `mode` | `reservation` | `group` | `reservations` |
| `GlobalNav` / `PageFrame` | `cartMode` | `reserve` | `hold` | `reservations` |
| `PoliciesAgreement` | `flow` | `reserve` | `group` | `reserve` |

¹ The **Book Reservation** landing/widget uses `mode="reservations"` — the widget
has only two toggle options (`reservations` = *Book Reservations*, `group` =
*Hold Rooms for Group or Team*); the single-vs-multi distinction is a checkout
concern, not a widget one.

**Rules of thumb:** cart/confirmation/room components speak
`reserve | hold | reservations`; the checkout page speaks
`reservation | group | reservations`; the widget/landing speak
`reservations | group`. When in doubt, grep the component's `defineProps`.

## Design tokens & theming

- **Always use DS tokens**, never hardcoded hex. Tokens are CSS custom properties
  named `--ds-*`:
  - Palette ramps: `--ds-palette-<hue>-<step>` (e.g. `--ds-palette-navy-900`).
  - Semantic roles: `--ds-color-text`, `--ds-color-text-subtle`,
    `--ds-color-surface`, `--ds-color-border`, `--ds-color-border-bold`,
    `--ds-color-background-brand-bold`, `--ds-color-background-danger-bold`,
    `--ds-color-text-danger`, `--ds-color-text-success`.
  - Scales: `--ds-radius-*`, `--ds-duration-fast`, `--ds-ease-standard`, `--ds-shadow-*`.
- **Primary brand = Navy** (`#01113E`). Reach for `--ds-color-background-brand-bold`
  for actions and `--ds-color-text-brand` for brand headings.
- Page canvas uses `--ds-palette-slate-100`; foreground cards use
  `--ds-color-surface` with a `--ds-color-border` outline (the checkout/account pattern).

## Component conventions

- **Vue 3 SFC, `<script setup>`.** Props via `defineProps`, events via
  `defineEmits`, imperative API via `defineExpose`.
- **Quasar is globally registered** — use `<q-*>` tags directly (no per-file
  import). Plugins available: `Notify`, `Dialog`, `Loading` (see `preview.js`).
- **Scoped styles**, BEM-ish naming: one **prefix per component**, then
  `prefix__element` / `prefix--modifier` (e.g. `.hc__footer`, `.rcr__cta`,
  `.pmf__grid`). Teleported content (menus, tooltips, notifies) needs an
  **unscoped** `<style>` block.
- **State sync** patterns in use: `v-model:<name>`, parent-owned reactive lists
  passed as props + `remove`/`update` events, and `defineExpose({ … })` for
  imperative APIs like `clear()`.
- **Modals & panels are primitives.** Centered dialogs use **`DsModal`**;
  slide-overs use **`DsSidePanel`**. Don't hand-roll overlay chrome.
- **Amenities come from one catalog** (`src/lib/amenities.js`) — reference by
  `key` and render via `Amenity`. Don't hardcode amenity lists.
- Match the **comment density and idiom** of the file you're editing.

## Story conventions

- One `*.stories.js` per component. `title: 'Category/Name'` decides placement;
  add `tags: ['autodocs']` for a generated Docs tab.
- **Variants are named exports** (`export const GroupBlock = …`). Full-page
  components set `parameters: { layout: 'fullscreen' }`.
- **Flow-first hierarchy:** each experience flow has a top-level
  `Book Reservation` + `Group Block` story and a `Components` folder; inside
  Components, Book Reservation and Group Block **sub-folders** list their pieces
  in the same order (Review Order → Contact Info → … ).
- Live/interactive demos use a `render()` with a `setup()` returning reactive
  state.
- Sidebar order is **explicit** in `preview.js` `storySort` — anything after
  `'*'` sorts last (e.g. `Manage Booking`).

## Building the localized prototype

The library is currently a **prop-driven** component gallery — pages take fixed
data as props and emit events; there is **no router and no shared store**. To wire
a clickable end-to-end journey you add three things on top of the existing
components. **Prototype mechanism is deliberately undecided** — the same
building blocks support all of these:

- **In-Storybook** — a `Prototype/User Journey` story that mounts the pages and
  swaps them on the emitted events (`choose`, `select`, `back`, step `next`,
  `confirm`). Zero new tooling; lives beside the docs.
- **Separate Vue app** — a `src/app/` entry with `vue-router` importing these
  components as a library. Closest to production.
- **Figma** — assemble screenshots of these stories into a click-through via the
  Figma MCP. No code.

Whichever you pick, you need **navigation**, **shared booking state**, and
**shared data**:

1. **Shared booking state (recommended: a single reactive object).** One source
   of truth threaded through every page: `{ mode, event, hotel, dates, guests,
   rooms[], cart, contact, payment }`. A plain reactive module
   (`src/lib/journeyState.js`) with `provide`/`inject` is enough for a prototype;
   promote to Pinia only if the app grows. **`mode` lives here** and every page
   reads it (remember the per-component name mapping above).
2. **Navigation.** Map the emitted events to transitions — the Build Spec lists
   the exact event per stage. In-Storybook: a `ref` of the current screen. App:
   `vue-router` routes `/ → /browse → /hotel/:id → /checkout → /confirmation`.
3. **Shared data (recommended: a fixtures layer).** Today mock data is written
   **inline per story**, which drifts between screens. For a coherent journey,
   extract it to `src/lib/fixtures/` (`events.js`, `hotels.js`, `rooms.js`,
   `guests.js`) as the single dataset both the stories and the prototype read —
   so the hotel you pick in Browse is the hotel you see in Details and Checkout.
   The existing `_fixtures.js` / `_rooms-sample.js` / `lib/amenities.js` are the
   starting seeds. (`references/` is gitignored — screenshots only.)

The **User Journey & Build Spec** doc specifies each stage's inputs, required
fields, edge cases, and the event it emits — build the prototype from that.

## Adding a component (checklist)

1. Create the SFC in `src/components/` (or the right `<area>/`).
2. Use DS tokens + the prefix naming convention; reuse a shared block if one fits.
3. Respect the **mode cheat-sheet** — accept the same prop name/values as sibling
   components in that flow.
4. Add `src/stories/<category>/Name.stories.js` with `title`, `tags: ['autodocs']`, variants.
5. If it's a new category, add it to `storySort` in `.storybook/preview.js`.
6. Verify with `pnpm build-storybook` (the deploy gate).

## Gotchas

- **`<filter>` is a native SVG tag** — register the Filter under a distinct tag
  (e.g. `<browse-filter>`), or Vue renders an empty SVG instead of your component.
- **Mode-name drift** — the same journey uses `reserve`/`reservation`,
  `hold`/`group` on different components. Check the cheat-sheet.
- **Notify/menus/tooltips are teleported** — style them from an **unscoped**
  `<style>` or global `app.scss` (e.g. `.ds-toast`, `.pmf__tooltip`, `.cr__tooltip`).
- **Don't set `position` on a notification** (`.ds-toast`) — Quasar's leave
  transition sets `position:absolute`; an override breaks the exit animation.
- **Maps need an API key** — `HotelMap` degrades to a placeholder; never assume a key exists.
- **`hold` never shows totals/taxes** — the group block is held, not charged. No
  payment step, no protection, nightly cost only.
- **`references/` is local-only** (gitignored) — design screenshots live on disk.
- **Run from a git branch**, not `main`; deploy happens automatically on merge to `main`.
